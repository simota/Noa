# UX Design: theme-settings-v2 リッチ化9要素

## Metadata

- **対象spec:** `docs/specs/theme-settings-v2.md`(R-25〜R-33のうちUI新設を伴う9要素)
- **前提:** v1の視覚言語(AppKitネイティブカード+TUIフォールバック、`crates/noa-app/src/chrome.rs`のトークン)は確立済み。本ドキュメントは新しい視覚言語を発明せず、既存トークン・既存レイアウト座標系への増分としてのみ設計する。
- **設計原則:** (1) マウス非対応・キーボード駆動が仕様(`handle_theme_settings_key`が唯一の入力経路)。(2) 保全制約5(3描画同期点の単一情報源)— 新規表示要素は`RowDraft::display_value`/`settings_row_display_value`と同格の共有関数を経由し、wgpu経路とネイティブ経路をフォークしない。(3) 単一アクセントhue原則(`chrome.rs`の`CHROME_ACCENT`ドキュメントコメント: 「chrome/overlay選択/pane focusが同一hueで統一されている」)— 新規の色相を増やさない。

## 0. グラウンディング中に発見した2件の事実訂正

実装前に解消しておくべき、spec本文(`theme-settings-v2.md`)の記述と実コードの不一致。

1. **R-27の「`DEFAULT_MINIMUM_CONTRAST`と同値」は誤り。** `noa-render/src/theme.rs:13`の実値は `DEFAULT_MINIMUM_CONTRAST: f32 = 1.0`(「自動調整なし」を意味する値、`noa-config/src/lib.rs:30`のコメント参照)であり、4.5ではない。4.5はテストコード(`theme.rs:466-472`)でWCAG AA閾値として使われている数値に過ぎない。**本設計は4.5:1をこの機能専用の独立したリテラル定数として扱う**(`DEFAULT_MINIMUM_CONTRAST`を参照・比較しない)。
2. **ネイティブ側フッター(`macos_overlay/model.rs:229-236`)は現在モード非依存で、Theme モードでも常に`←→ adjust`を含む文言を表示している。** wgpu側(`theme_picker_overlay_text` vs `settings_rows_overlay_text`)は既にモード別に正しく出し分けている。本設計のフッター刷新(§2)はこの不一致を解消する形で行う(保全制約5の観点でも「両経路が同じ値を返す」の欠落を埋める)。

## 1. 一致件数表示(R-26)

### 内容・書式
- `{highlighted_index + 1} / {filtered_len}`(例 `3 / 12`)。分母は**現在有効な絞り込み後**の件数 — fuzzyテキストのみでなく、お気に入りフィルタ(§4)・明暗フィルタ(§5)がONの場合はその適用後の件数を表す(574固定ではない)。この定義はR-26本文の`filtered_len()`の定義とそのまま整合する。
- 0件時: `0 / 0` ではなく **`No matches`**(既存コードベースにこの空状態文言の先例は無いため新規制定。Title Caseは既存行ラベル`Font Size`等と同じ慣習に合わせる)。

### 配置
- **wgpu**: 既存の filter 行(`LIST_TOP_ROW - 1` = 行3、`LIST_COL`起点で`/{filter}`を表示中)の**右端に追加**。list列幅(`LIST_WIDTH` = 28)の右端(列27)に右寄せ。`/{filter}`が長い場合でも重ならないよう、件数側を右寄せ固定・filter側は残り幅で打ち切り表示にする(既存の`cols.saturating_sub(...)`系の打ち切りパターンを踏襲)。
- **ネイティブ**: filter ラベル(`y=64, x=pad, w=col_split-pad-8`)と同じ行の右端に新規ラベルを追加(`x = col_split - pad - 60, w = 60`程度、右寄せ、`mono_digit_font`で桁が揃うようにする — スクラブ中に桁の増減でテキストが左右にジッタしないようタブラー数字を使う)。
- **色**: muted(既存filter行と同じ)。強調要素ではないため常時アクセントにしない。

## 2. キーヒントフッター(全モード共通刷新)

### 方針
新規操作(Tab再オープン・お気に入りトグル)が増える一方、footer1行に全キーを詰めると認知負荷が上がる(パレットスキルの原則)。**高頻度・非自明な操作のみ footer に常設**し、低頻度операция(属性フィルタ切替・お気に入りのみ表示)は該当コントロール自体の隣に局所ヒントを出す(近接性 — 目がすでにある場所にヒントを置く方が、footerに詰め込むより発見性が高い)。

### Theme モードの footer(新規)
```
↑↓ Navigate   ⇥ Settings   ⌃F Favorite   ⏎ Save   Esc Cancel
```
### Settings モードの footer(新規)
```
↑↓ Navigate   ⇥ Theme   ←→ Adjust   ⏎ Save   Esc Cancel
```
- `⇥`(Tab)セグメントを新規追加: これまでTabは無反応(`toggle_section`空実装)だったため、R-25でTabに新しい意味が付いた今、**footerで明示しないと発見不可能**な変更点である。行き先モード名を直接書く(`⇥ Settings` / `⇥ Theme`)ことで「Tabを押すと何が起きるか」を予測可能にする。
- `⌃`(Ctrl)グリフはこのリポジトリの既存コード(`command_palette.rs`の`keybind_symbols`、`\u{2303}` = ⌃)がすでに採用しているmacOS標準修飾キー記号セットをそのまま流用する — 新しい記法を発明しない。
- 属性フィルタ(⌃D)とお気に入りのみ表示(⌃⇧F)のヒントはfooterに含めない。§5のチップ行自体に局所キャプションとして表示する(下記)。
- **幅の実装上の懸念**: `THEME_SETTINGS_COLS = 56`固定幅のwgpu経路では、Theme モードのfooter文言はこの列数ぎりぎりに収まる長さになる。狭いペインでは`cols.min(pane_cols - 4)`によりさらに縮む可能性があるため、**優先度順(削る場合はこの順で末尾から削る): Esc Cancel > Enter Save > ↑↓ Navigate > (Settingsのみ)←→ Adjust > ⇥ 切替 > ⌃F Favorite**。ネイティブ側はラベル幅が可変なので実質問題にならない。
- **エラー状態は変更なし**: `commit_error`がある間はfooter全体をdanger色のエラー文言に差し替える既存挙動(`Tone::Danger`)をそのまま維持する。
- **実装契約**: wgpu/ネイティブ両方が同じ「モード別footer文言を返す一つの共有関数」を呼ぶこと(§0-2で指摘した既存の不一致を再発させないため)。

## 3. コントラスト比表示+低コントラスト警告(R-27)

### 数値書式
- `Contrast 4.8:1`(小数点1桁、`{ratio:.1}:1`)。`noa_render::theme::contrast_ratio(default_fg, default_bg)`をそのまま呼ぶ(新規計算ロジックなし、既存関数の再利用のみ — R-27本文の制約通り)。

### 警告の視覚表現 — spec本文への修正
R-27原文は「色/アイコンいずれか」と両者択一で書かれているが、**これはWCAG 2.2 SC 1.4.1(色の使用)に抵触する**(色だけの警告は非color-visionユーザーに伝わらない)。本設計では **アイコン+文言変化の両方を必須** とし、色はあくまで補助とする:
- 通常時(≥4.5): `Contrast 4.8:1`(muted色、アイコンなし)
- 警告時(<4.5): `⚠ Contrast 2.1:1 — low`(danger色 = `crate::chrome::palette().dot_red`と同系統、`⚠`アイコン付き、末尾に`— low`という文言そのものを追加して色に依存しない伝達にする)
- 閾値4.5は本機能専用のリテラル定数として実装する(§0-1参照、`DEFAULT_MINIMUM_CONTRAST`は参照しない)。

### 配置
サンプル欄の「実際に何と何のコントラストを見ているか」が一目でわかるよう、既存のfg/bg swatch(semantic swatches列、`Foreground`/`Background`)の**直下**に置く。既存のtruecolorランプの下、新設のサンプル文行(§7)よりさらに下 — 「色見本→文脈内での見え方→その要約数値」という抽象→具体→結論の縦順。
- **wgpu**: `LIST_TOP_ROW + 4`行目(既存: ANSI 2行+semantic行+truecolor行 = `LIST_TOP_ROW`〜`+3`の直後)、`SAMPLE_COL`起点。
- **ネイティブ**: `semantic_top + sw + gap + 4.0`(truecolor ramp帯)からさらに`+14pt`下、`sample_x`起点、11pt、`system_font`(既存ラベルと同じファミリー)。

## 4. お気に入り星マーク(R-29)

### リスト行内の表示
- 星は**お気に入り済みの行にのみ表示**(`★`, U+2605)。未お気に入りの行には`☆`のような outline を常設しない — 574行すべてに空アイコンを出すと視覚ノイズが増える(認知負荷原則)。「星がある=お気に入り、無い=違う」の二値だけで十分。
- **位置**: 行の右端(行名の後、右寄せ)。行名の左端はfuzzyハイライト対象なので触れない。
  - wgpu: 各テーマ行の`LIST_COL + LIST_WIDTH - 1`(list列の最終列)に1文字。
  - ネイティブ: 行ラベル矩形の右端に`14×14pt`の小ラベルを追加(`x = col_split - pad - 16`程度)。
- **色**: accent(`CHROME_ACCENT`)。新しい色相(金/黄)を導入せず、既存の単一アクセントhue原則を維持する(§本文冒頭の設計原則3)。ハイライト行のaccent文字色と隣接しても、アイコン vs 文字という形状差で判別できるため衝突しない。

### トグルキー割当
- **`⌃F`(Ctrl+F)** — ハイライト中の行のお気に入りをトグル。Ctrl修飾はfuzzy文字入力(バレ文字キー)と衝突しない(Ctrl+文字はfuzzyフィルタのテキストバッファに挿入されない)。ネイティブ側のfilterラベルは実際には非編集の`NSTextField`(`make_label`、フォーカス不能)であることを確認済みのため、Cocoa標準のEmacs風`Ctrl+F`(カーソル前進)テキスト編集バインドとは衝突しない — 全キー入力はwinit経由でRust側`handle_theme_settings_key`が排他的に処理する。
- **`⌃⇧F`(Ctrl+Shift+F)** — 「お気に入りのみ表示」フィルタのトグル(§5のチップ行に配置)。同じ`F`のニーモニックをShiftで「行単位の操作→リスト全体のビュー切替」に拡張する一貫したペアとして設計。

### フィルタON状態の表示
§5のチップ行に `★ Favorites` (ON: accent文字色+`selected_bg`同等の背景ウォッシュ) / `☆ Favorites`(OFF: muted、背景なし)という1つのチップとして表示する。

## 5. 明暗属性フィルタチップ(R-30)

### UI形式
3値セグメント: `All · Dark · Light`。**新しいプリミティブ(角丸ピル背景)は発明しない** — 既存の「選択中の行」表現(ネイティブ: `tint_layer`による`selected_bg`背景ウォッシュ+accent文字色、wgpu: accent文字色+マーカー)をそのまま再利用し、アクティブなセグメントに同じ処理を適用する。

- **wgpu**: `All  [Dark]  Light` — アクティブセグメントのみ `[` `]` で囲みaccent色、非アクティブはmuted色プレーンテキスト(既存のfooter/hint文言と同じ「色のみのテキスト」の慣習を踏襲、新しい罫線プリミティブを使わない)。
- **ネイティブ**: 3つの小ラベルを横に並べ、アクティブなものだけ背景に`selected_bg`の`tint_layer`(既存の選択行と同じ`6.0`半径)を敷く。

### 配置
§1の一致件数行のさらに下に新設する「フィルタチップ行」に、お気に入りチップ(§4)と同居させる。**左**: 明暗チップ `All · Dark · Light`。**右**: お気に入りチップ `☆/★ Favorites`(§1の一致件数と同じ右寄せパターンを踏襲、視覚的なリズムを一致行と揃える)。
- wgpu: 新設行を `LIST_TOP_ROW` に挿入し、既存のテーマ一覧を1行分下にずらす(`LIST_TOP_ROW`実質+1相当、`LIST_ROWS`を8→7に削るか`THEME_SETTINGS_ROWS`を25→26に広げるかは実装判断 — カード高さの安定性を優先するなら前者)。
- ネイティブ: `list_top`(現行86pt)の手前に20pt分の行を追加し、`list_top`を106ptへ、カード高さ算出式(`needed()`)にこの+20ptを組み込む(ネイティブは動的にカード高を計算しているため、リストを削るより高さを20pt伸ばす方が自然 — 固定セル格子のwgpuと動的レイアウトのネイティブで縮退戦略が異なってよい、両者は既に同じ理由で異なる)。

### キー割当
- **`⌃D`(Ctrl+D)** — All → Dark → Light → All を巡回。ニーモニックは弱いが(D=display mode)、`⌃L`はターミナルの「画面クリア」の強い連想と衝突するため避けた。チップ行の隣に`⌃D cycle`という局所キャプションを添えて発見性を補う(footerには載せない、§2参照)。

## 6. コミット後Undoトースト(R-31)

### 既存トーストシステムとの関係
既存の resize トースト(`state.resize_overlay: Option<(String, Instant)>`、`RESIZE_OVERLAY_DURATION = 750ms`)は**中央固定・矩形固定サイズ・単一行ラベル・スクリムなし・ボタンなし**という情報提示専用のピル(`draw_toast_card`/`rebuild_toast`)。Undoトーストはこの視覚言語をそのまま流用し、新しい見た目を作らない。実装上は`resize_overlay`を`ToastKind`でタグ付けして汎用化する(spec本文L2の記載通り)。
- `commit_theme_settings`はコミット成功時に**セッションを閉じる**(`self.theme_settings.take()`)ため、Undoトーストはオーバーレイが完全に閉じた**後**、通常のトーストと同じ画面中央のペイン上に表示される — 位置的な競合は発生しない(resizeトーストと同じスロットを使う設計は妥当)。

### 文言(Enter Save 成功直後)
- **Theme モードでコミットした場合**: `Theme set to "{name}" · ⌘Z to undo`
- **Settings モードでコミットした場合**: `Settings saved · ⌘Z to undo`(複数行が同時に変わりうるため、Themeのように個別名を列挙しない)
- 既存の`Chrome/tabs update on Save`・`Failed to save settings: {err}`と同じ簡潔な英語トーン(句読点なし、動詞短縮)に合わせた。

### キー割当
- **`⌘Z`(Cmd+Z)** — システム標準のUndo慣習に一致する最も予測可能な割当。既存の`KeybindEngine::default()`一覧にcmd+zは未使用(確認済み)。オーバーレイが閉じた後に効くグローバルコマンドとして新設する(`commit`と同じ書込み関数を直前スナップショットの値で再度呼ぶだけ — 新しい書込み経路を作らない、spec本文R-31の制約通り)。
- Undoトーストの表示時間が過ぎたら`⌘Z`は無音でno-op(「トーストが消えた=undo可能な窓が閉じた」というメンタルモデルをそのまま反映。追加のモーダルや「元に戻せません」通知は出さない — magiスコープの対話コスト最小化方針に合わせる)。

### 表示時間
resizeトーストの750msは「連続ドラッグ中に何度も出る」用途のため意図的に短い。Undoトーストは**1回きりの意思決定**を要求するため、同じ750msでは短すぎる(読む→判断する→物理的にキーを押す、の時間が確保できない)。macOSの他アプリの取り消しトースト慣習(Mail「送信取り消し」等が数秒〜十数秒)を踏まえ、**6000ms(6秒)**を推奨値とする(spec本文のOpen Questionsが未確定としていた値への回答)。resizeトーストと表示が重なった場合は新しい方(Undo)が古い方を即座に置き換える(spec本文L2エッジケース記載通り)。

## 7. プレビュー代表サンプル行(R-33)

### 設計方針
現行の`sample_swatches`(16 ANSI色ブロック+fg/bg/cursor/selectionブロック+truecolorランプ)は**そのまま残す**(正確な色参照として機能を維持)。その**直下**に、同じ色データを実際のテキスト行として再構成した3行を追加する — 「色見本(抽象)」の下に「文脈の中でどう見えるか(具体)」を積む。新しい色導出ロジックは追加しない(既存`Swatch`列挙値をそのまま再利用、R-33の制約通り)。

3行の内容(すべて等幅、サンプル欄の幅いっぱい):
1. **通常テキスト行**: `default_fg` on `default_bg`。文言: `Sample text on background`
2. **強調(ANSI)テキスト行**: 一般的なログ配色の連想が効く4語を、ANSI基本色スロット(1=red, 2=green, 3=yellow, 4=blue)の前景色で着色し、`default_bg`背景に乗せる。文言: `error` (red) `warning` (yellow) `info` (blue) `ok` (green) — 実際のシェル/ログ出力で見る配色パターンを模し、単なる色チップより実用的な確認になる。
3. **選択行**: `default_fg` on `selection_bg`(`ThemeDef`には選択専用の前景色が無いため、既存`sample_swatches`の`Selection`swatchと同じ制約を踏襲し、既定fgをそのまま使う)。文言: `Selected text`

### 配置
サンプル欄、truecolorランプの直下(§3のコントラスト表示のさらに上)。各行は1行7〜8文字程度の高さ(wgpu: 1テキスト行、ネイティブ: `system_font(12.5)`ラベル1行)。3行合計でも新設の空きスペース(現状170pt以降がネイティブでは未使用、wgpuでは行12〜23が未使用)に十分収まる。

### 3描画同期点の契約
wgpu(`theme_picker_overlay_text`)とネイティブ(`theme_settings_view_model`)の両方が、この3行を生成する**単一の共有関数**(例: `sample_text_lines(theme) -> [(String, Rgb, Rgb); 3]`、`sample.rs`に追加)を呼ぶこと。どちらか一方だけをフォークしない(AC-48が明示的に検証する契約)。

## 8. Tab切替時の視覚的連続性(R-25)

### 設計目標
Tabは「新しいモーダルを開く」動作ではなく「同じセッションの見せ方を変える」動作である(R-25のアーキテクチャ上の前提: `preview_theme`もライブ適用済みの値も一切変更しない)。見た目もそれを裏切ってはならない — 初回オープン時のフェードイン(`DUR_FAST` = 120msの透明度0→1 Tween、`app/render.rs`で使用中)を**Tab切替のたびに繰り返すと「毎回開き直された」ように見え、連続性の目標に反する**。

### 挙動
- **スクリム/不透明度は変化させない**(常時フル不透明のまま)。初回オープン専用のフェードインTweenはTab切替では発火させない。
- **カード枠のみサイズ遷移**: Theme モードとSettings モードでカード高さが異なる(Theme はテーマ数依存の可変高、Settings は16行固定)。中心点を固定したまま、旧モードのサイズから新モードのサイズへ`DUR_FAST`(120ms、既存定数を流用— 新しい値を作らない)でイージング遷移させる。
- **中身の切替は即時**(t=0で新モードのラベル/リストに差し替える。フレーム中に両モードのデータを同時に保持する必要がある「コンテンツ側のクロスフェード」は行わない — 実装コストに見合わない上、セッションは元々どちらか一方のモードのデータしか保持していない)。
- **バッジ(`Chrome/tabs update on Save`)・プレビュー中の実色**: `gpu.preview_theme`は変更されないため、切替前後で完全に同一の値を表示し続ける。これを保証するのはR-25のデータ層の責務だが、**視覚的にも「バッジが一瞬消えて再度出る」ような明滅を起こさないこと** — バッジの表示条件(`badge_visible()`)がTab遷移そのものによって偽の再計算トリガーにならないよう、既存の判定条件をそのまま素通しする。

## 9. マイクロコピー一覧(全件・英語)

既存パネルの文言トーン(句読点最小限・Title Caseは行ラベルのみ・hint/toastは簡潔な短文)に合わせて統一。

| 用途 | 文言 | 備考 |
|---|---|---|
| 一致件数(通常) | `{n} / {m}` | 例 `3 / 12` |
| 一致件数(0件) | `No matches` | 新規制定(既存に前例なし) |
| footer(Theme, 通常) | `↑↓ Navigate   ⇥ Settings   ⌃F Favorite   ⏎ Save   Esc Cancel` | §2参照、幅超過時は末尾から間引く |
| footer(Settings, 通常) | `↑↓ Navigate   ⇥ Theme   ←→ Adjust   ⏎ Save   Esc Cancel` | 同上 |

## Addendum A — Risk Gate (echo) 反映 2026-07-11

### A-1. ⌃⇧F の局所ヒント(echo実装前修正#3)
§5のチップ行・右側のお気に入りチップに、明暗チップの`⌃D cycle`と対称の局所キャプションを追加する:
- 表示: `☆ Favorites ⌃⇧F`(OFF) / `★ Favorites ⌃⇧F`(ON)。キャプション部`⌃⇧F`は`⌃D cycle`と同じmuted色・同サイズ。
- wgpu/ネイティブとも、チップ本体ラベルの右に半角スペース1つで連結した同一文字列として描画してよい(新プリミティブ不要)。
- footerには引き続き載せない(§2の方針維持)。

### A-2. フィルタ切替時のハイライト/プレビュー挙動(echo実装前修正#4)
属性フィルタ(⌃D)・お気に入りフィルタ(⌃⇧F)の切替、およびTab carryover適用によってfilteredが再計算されたとき:
1. **ハイライト中テーマが新しいfiltered内に残っている場合**: ハイライトはそのテーマを追跡する(インデックス再計算)。preview不変。
2. **除外されたが一覧が非空の場合**: `preview_theme`は変更しない(AC-16「直前プレビュー維持」パターンの拡張)。ハイライトは先頭(index 0)へ移動するが、`highlight_moved`をリセットして**ユーザーが明示的に↑↓で動かすまで新previewを発火しない**。「フィルタを切り替えただけでプレビューが勝手に変わる」ことを構造的に禁止する。
3. **一覧が空の場合**: 既存AC-16の挙動(一覧空・直前preview維持)をそのまま適用。
